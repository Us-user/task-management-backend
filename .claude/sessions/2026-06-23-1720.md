## Goal
Phase 0 — build the Fastify skeleton: `src/app.ts` (instance + plugin registration seam)
and `src/server.ts` (bootstrap + listen), replacing the placeholder source root.

## Done This Session
- **`src/app.ts`** — new `buildApp(opts)` async factory. Returns a configured-but-not-listening
  Fastify instance so tests can drive it via injection without binding a port. Wires pino
  logging (`defaultLogger()`: `pino-pretty` in dev, structured JSON in prod, `LOG_LEVEL`-aware),
  `trustProxy: true`, request logging on. Includes a basic `GET /health` liveness route
  (`{ status, uptime }`) and clearly-marked seams for base plugins (helmet/cors/rate-limit/
  swagger/error-handler) and the `/api/v1` module tree.
- **`src/server.ts`** — replaced the `PLACEHOLDER` export with a real bootstrap: builds the app,
  reads `HOST`/`PORT` (defaults `0.0.0.0:3000`), installs `SIGINT`/`SIGTERM` handlers that call
  `app.close()` (drains in-flight requests, runs plugin `onClose` hooks), and `app.listen()`
  with error-exit handling.
- **Verification:** `npm run typecheck`, `npm run lint` (eslint + prettier), `npm run build` all
  green. Booted the compiled server on port 3999 → `GET /health` returned
  `{"status":"ok","uptime":...}`; SIGTERM shut it down cleanly (exit 0).

## Decisions Made
- **Env handling kept minimal in `server.ts`** (plain `process.env` + defaults for HOST/PORT/
  LOG_LEVEL) rather than pre-building Zod validation — the full validated `src/config/` is the
  next roadmap task and owns that concern. Avoids doing it twice.
- **`buildApp` is async** (returns `Promise<FastifyInstance>`) from the start, so later plugin
  `register` calls and `app.ready()` needs don't force a signature change.
- **`/health` left as a basic liveness check** — real readiness (DB ping) is deferred until
  Prisma is wired. The dedicated "`GET /health` endpoint" roadmap item stays unchecked because
  it intends the full liveness/readiness probe; only the skeleton stub exists today.
